# Personas

본 문서는 이사 갈 동네 리포트 서비스의 핵심 페르소나를 정의합니다. 각 페르소나는 Standard 상세 수준(이름·역할·목표·관심사·기술 친숙도·시나리오)으로 작성되었습니다.

---

## P-01. 이사 예정자 (공통)

**역할**: Primary User — 이사 후보지를 비교하여 의사결정을 내리는 사용자

**핵심 목표**:
- 후보 동네를 객관적 데이터로 비교하여 합리적 의사결정
- 흩어진 정보(실거래가, 대기질, 시설 등)를 통합 리포트로 한눈에 확인
- 후회 없는 이사 결정

**관심사**:
- 주거비 적정성
- 생활 편의성과 인프라 접근성
- 환경 쾌적성
- 안전 인프라
- 데이터 신뢰성과 출처 투명성

**기술 친숙도**: 보통 (스마트폰·웹 기본 사용자, 부동산 앱 경험 있음)

**시나리오**:
이사를 3개월 내로 계획 중. 출퇴근, 예산, 생활 환경을 종합적으로 고려하여 3~5개 후보지를 비교하고 싶다. 광고성 매물 정보보다 공공 데이터 기반의 객관적 분석을 신뢰한다.

**대표 인용**: "광고에 휘둘리지 않고 데이터로 결정하고 싶어요."

## P-02. 1인 가구 이사 예정자

**역할**: User Sub-segment — 1인 가구 라이프스타일을 가진 이사 예정자

**핵심 목표**:
- 야간 귀가 안전, 출퇴근 편의, 생활 편의시설 접근성 중심으로 후보지 평가
- 합리적 주거비로 1~2인 거주에 적합한 면적대 확인

**관심사**:
- 지하철역·주요 대중교통 거점 접근성
- 24시간 편의시설(편의점·약국·응급의료) 접근성
- 1인 가구 적합 면적대(~59㎡) 실거래가
- 야간 안전 인프라(경찰서·CCTV 데이터 가용 시)

**기술 친숙도**: 높음 (모바일 우선 사용자, 다양한 부동산·생활 정보 앱 활용)

**시나리오**:
첫 직장 근처로 독립을 준비. 회사 지하철역 기준 30분 이내에서 후보지 3개를 비교하고 있다. 모바일에서 빠르게 후보지 등록·비교·삭제를 반복한다.

**대표 인용**: "야근 후 안전하게 귀가할 수 있는지가 가장 중요해요."

## P-03. 가족 이사 예정자

**역할**: User Sub-segment — 자녀 양육 중인 가족 단위 이사 예정자

**핵심 목표**:
- 자녀 교육 환경(학교 접근성, 학군) 우선 고려
- 가족이 거주 가능한 면적대(84㎡ 이상) 실거래가 확인
- 병원·공원 등 가족 친화 인프라 점검

**관심사**:
- 학교 거리·종류·수
- 공원·놀이터 접근성
- 가족형 면적대(84~135㎡) 실거래가
- 응급의료기관 접근성 (자녀 응급 대비)

**기술 친숙도**: 보통 (PC·모바일 혼용, 지도와 비교 테이블 선호)

**시나리오**:
초등학생 자녀를 둔 부모. 학교 통학과 병원 접근성을 우선 고려하며, 부부 둘이 검토할 수 있도록 비교 결과를 명확히 보고 싶다. 부동산 광고보다 공공 데이터 기반 정보를 선호.

**대표 인용**: "아이 학교와 병원이 가까운 곳을 데이터로 확인하고 싶어요."

## P-04. 환경/건강 민감 사용자

**역할**: User Sub-segment — 대기질·기상 민감도가 높은 이사 예정자 (비염·천식·고령자 동거 등)

**핵심 목표**:
- 후보지의 대기질 수준과 기상 리스크를 사전 점검
- 건강에 영향을 주는 환경 지표를 우선순위 가중치로 반영

**관심사**:
- PM10·PM2.5 평균값·변동성
- 폭염·한파·강수 패턴
- 가장 가까운 대기측정소 거리와 측정소 신뢰도
- 환경 가중치 프리셋 활용

**기술 친숙도**: 높음 (대기질 앱·기상청 앱 정기 사용자)

**시나리오**:
천식이 있어 대기질이 매우 중요. "환경 가중치 우선" 프리셋을 적용해 후보지 점수를 재정렬하고 비교한다. 측정소 위치까지 확인할 만큼 데이터에 민감.

**대표 인용**: "수치만 보면 안 되고 측정소가 어디 있는지까지 알아야 해요."

## P-05. 운영자 (Superadmin)

**역할**: Internal User — 시스템 최고 권한 운영자, 다른 admin 계정 관리 가능

**핵심 목표**:
- Admin 계정 생성·관리, 권한 통제
- 시스템 전체 가중치 기본값·프리셋 설정
- 보안과 안정성 정책 수립

**관심사**:
- 운영자 계정 생성·제거·역할 변경
- 점수 가중치 정책의 타당성
- 감사 로그 무결성
- 보안 정책 (비밀번호 정책, 토큰 만료 등)

**기술 친숙도**: 매우 높음 (개발/운영 백그라운드, REST API·로그 분석에 익숙)

**시나리오**:
초기 배포 시 superadmin 계정으로 로그인하여 admin 계정을 추가하고 가중치 프리셋을 등록. 이후 정책 변경이나 보안 이슈 대응을 담당.

**대표 인용**: "권한 분리가 명확해야 운영 사고를 막을 수 있죠."

---

## P-06. 운영자 (Admin)

**역할**: Internal User — 일상 운영을 담당하는 운영자, admin 권한

**핵심 목표**:
- 공공 API 연동 상태 모니터링과 장애 대응
- 캐시 갱신·관리
- 가중치 프리셋 운영(추가/수정), 단 superadmin이 잠근 항목은 변경 불가

**관심사**:
- API 정상 동작 여부 (정상/지연/실패/할당량)
- 캐시 적중률과 갱신 주기
- 사용자 리포트 생성 실패 사례
- 운영 효율 (수동 재시도, 캐시 수동 갱신)

**기술 친숙도**: 높음 (운영 도구·대시보드에 익숙)

**시나리오**:
매일 아침 Admin Panel에 로그인해 API 상태를 확인. 지연·실패가 있으면 수동 재시도, 캐시 상태 점검. 사용자 문의 대응을 위해 특정 지역 캐시를 수동 갱신.

**대표 인용**: "외부 API 한 곳이라도 장애 나면 즉시 알아차려야 해요."
